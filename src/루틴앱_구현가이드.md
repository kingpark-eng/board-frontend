# ADHD 루틴 앱 — 혼자 만들기 가이드

게시판(Spring Boot + React)을 만든 경험을 그대로 재활용한다.
핵심 원칙: **한 층씩, 만들 때마다 테스트.** 한꺼번에 다 만들지 않는다.

> 게시판과의 대응 관계 (이걸 옆에 두고 베껴 만들면 됨)
> ```
> Post           → Routine
> PostController → RoutineController
> PostService    → RoutineService
> PostRepository → RoutineRepository
> postApi.js     → routineApi.js
> PostListPage   → RoutinePage
> ```

---

## PHASE 0 — 가장 단순한 버전부터

처음엔 반복/날짜 기록을 빼고, **그냥 할 일 CRUD**만 만든다.
(매일/주간 반복, 완료 기록, streak는 PHASE 5에서 얹는다.)

이번 단계의 Routine은 이 필드만:
```
id, title, done(완료 여부), user_id
```

게시판의 Post와 거의 똑같다. 여기서 패턴을 한 번 더 굳히는 게 목적.

---

## PHASE 1 — 백엔드: 데이터 한 덩어리 (Entity → Repository)

### 1-1. Routine 엔티티
- [O] `entity/Routine.java` 생성
- [O] 필드: id, title, done, user(@ManyToOne)
- [O] `@Entity @Getter @NoArgsConstructor @Builder` — Post 보고 그대로
- 확인: User 엔티티 import 경로가 맞는지
- @AllArgsConstructor 추가함

### 1-2. Repository
- [O] `repository/RoutineRepository.java` 생성
- [O] `extends JpaRepository<Routine, Long>`
- [O] 내 루틴만 조회하는 메서드: `List<Routine> findByUserId(Long userId)`
- 확인: interface 인지(class 아님), JpaRepository import 맞는지

### ✅ PHASE 1 테스트
- 서버를 켠다 → 에러 없이 뜨는지
- DB에 `routines` 테이블이 자동 생성됐는지 확인
- (아직 API가 없으니 화면 테스트는 안 됨. 서버가 뜨면 OK)

확인완료.

## PHASE 2 — 백엔드: DTO

요청/응답 그릇을 나눈다. 게시판 PostRequest / PostResponse 패턴 그대로.

### 2-1. 요청 DTO
- [O] `dto/RoutineRequest.java`
- [O] 안에 `Create { title }`, `Update { title }` static class
- [O] `@NotBlank` 검증 붙이기

### 2-2. 응답 DTO
- [O] `dto/RoutineResponse.java`
- [O] 안에 `Detail { id, title, done }` static class
- [O] 엔티티 → DTO 변환 생성자 또는 from() 메서드

### ✅ PHASE 2 확인
- 컴파일 에러 없으면 OK (아직 호출 안 함)

## PHASE 3 — 백엔드: Service → Controller

### 3-1. Service
- [O] `service/RoutineService.java` (`@Service`)
- [O] 메서드 5개: 목록조회, 단건조회, 생성, 수정(또는 완료토글), 삭제
- [O] 생성/조회 시 "현재 로그인 유저"의 것만 다루기
  - 게시판 login에서 했던 것처럼 email로 User 찾아 연결
- 핵심: **남의 루틴을 못 건드리게** 본인 것인지 확인하는 로직

### 3-2. Controller
- [O] `controller/RoutineController.java` (`@RestController`)
- [O] `@RequestMapping("/api/routines")`
- [O] 엔드포인트:
  ```
  GET    /api/routines           내 루틴 목록
  POST   /api/routines           루틴 추가
  PATCH  /api/routines/{id}/done 완료 토글
  DELETE /api/routines/{id}      삭제
  ```
- [X] `@AuthenticationPrincipal`로 로그인 유저 받기 (게시판과 동일)

### 3-3. SecurityConfig 경로 추가
- [O] `/api/routines/**` 는 인증 필요(authenticated)로 두기
  - 루틴은 본인만 보는 거라 PUBLIC 아님
-> .requestMatchers("/api/routines/**").authenticated()로 설정함.

### ✅ PHASE 3 테스트 (중요)
프론트 만들기 전에 **API부터 검증**한다.
- [O] 브라우저 개발자도구 또는 Postman으로 직접 호출
- [O] 먼저 로그인해서 토큰 얻기 → 헤더에 `Authorization: Bearer {토큰}`
- [O] POST로 루틴 하나 만들어보기 → DB에 들어갔나 확인
- [ ] GET으로 목록 나오나 확인
=> 403, permit으로 변경 후, 500에러 발생.
- [ ] 토큰 없이 호출하면 401 나오나 확인 (인증 동작 검증)

여기까지 되면 백엔드 완성. 프론트는 이걸 화면에 붙이는 것뿐.

- [ ] 새로고침해도 유지되나 (DB 저장 확인)
- [ ] 로그아웃 상태에서 /routines 가면 어떻게 되나

🎉 여기까지가 동작하는 MVP. 이것만으로도 쓸 수 있다.

## PHASE 5 — 반복 루틴 + 날짜 기록 (MVP 된 다음에)

MVP가 돌아간 다음에만 손댄다. 안 그러면 복잡해서 막힌다.

### 핵심 개념: 규칙과 기록 분리
```
Routine      "매일 약 먹기"   ← 규칙 하나
RoutineLog   날짜별 완료 여부  ← 날짜마다 따로
```
- [ ] Routine에 repeatType(DAILY/WEEKLY), weekDays, remindAt 추가
- [ ] RoutineLog 엔티티 추가 (routine, logDate, done)
- [ ] "오늘 해야 할 루틴" 조회 로직 (오늘 요일에 맞는 것만)
- [ ] 완료 토글 = 오늘 날짜의 RoutineLog를 만들거나 뒤집기
- [ ] (선택) 연속 달성(streak) 계산해서 보여주기

### ADHD UX 얹기 (여유 될 때)
- [ ] 헤더에 "오늘 할 루틴" 상시 표시
- [ ] 완료 시 시각적 보상 (체크 애니메이션)
- [ ] 미완료가 눈에 띄게 (색 강조)

## HOOK의 종류
- 상태 관리 — useState. 컴포넌트가 기억해야 하는 값(입력값, 토글 상태, 로딩 여부 등)을 담습니다. 값이 바뀌면 화면이 다시 그려집니다.
- 부수 효과 — useEffect. API 호출, 구독, 타이머처럼 렌더링 "이후에" 일어나야 하는 일을 처리합니다. 메모리에 정리해두신 의존성 배열 규칙([]=마운트 시 한 번, 없으면 매 렌더, 값 넣으면 그 값 바뀔 때)이 바로 이 훅 얘기죠.
- 전역 상태 공유 — useContext. useAuth가 내부적으로 이걸 씁니다. props를 계속 내려보내지 않고 어디서든 인증 상태를 꺼내 쓰게 해줍니다.
- 커스텀 훅 — useAuth처럼 use로 시작하는 직접 만든 훅. 여러 컴포넌트가 공유하는 로직을 하나로 묶어 재사용합니다.
